Two ways to use

◎ Agent via MCP

A stdio MCP server with 9 tools. Coding agents (Claude Code, Codex, Cursor, Windsurf, OpenCode) query durable code memory instead of re-reading files on every task.

Register once with chaos setup, then ask the agent to analyze and query. Skills + hooks also ship for Claude Code and Cursor.

▶ Raw CLI

The chaos binary: analyze, add, stats, query, feature-context, impact, graph, obsidian, refresh. For humans and scripts doing setup, one-off indexing, debugging, or agentless operation.

chaos analyze /path/to/repo
chaos add /path/to/repo
chaos stats /path/to/repo
chaos query /path/to/repo "where is auth?"
chaos impact /path/to/repo "add OAuth2 login"

◆ Generated feature-website

A self-contained dark HTML feature page with interactive graph/story/code navigation plus a machine-readable manifest — for sharing how a feature works and seeding future agent context from the embedded manifest.

chaos feature-context /repo "auth flow" \
  --output-html auth.html
How it works

From source to durable memory

Every file in the repository is parsed by a real AST extractor (no regex). The extracted graph and embedding-indexed chunks live in Postgres+pgvector and survive restarts. Queries use hybrid retrieval — semantic + keyword — against that persistent store.

Prerequisites

Postgres + pgvector

The bundled docker-compose.yml starts pgvector/pgvector:pg16 on host port 54329:

docker compose up -d

Default connection string:

postgres://chaos:chaos@localhost:54329/chaos_substrate

Your data stays in your own database. Nothing is sent to a third party beyond your chosen embedding provider.

A real embedder

Choose one — analysis will refuse to run without a working provider:

  • Ollama (default) — nomic-embed-text, 768 dims, http://localhost:11434. No API key needed. The example config defaults to Ollama; run ollama pull nomic-embed-text before indexing. See docs/OLLAMA_SETUP.md.
  • OpenAI (alternative) — text-embedding-3-small, 1536 dims. Uncomment the open_ai block in your config and set OPENAI_API_KEY. Only chunk text is sent.

Rust toolchain

Build the release binary once and point all editor/MCP config at it:

cargo build --release
# produces: target/release/chaos
⚠ Never use cargo run in MCP/editor config. The release binary is the stable, fast entry point. cargo run is fine only for one-off CLI commands during development.
Quick start

Bootstrap in five steps

1
Clone and copy config

The example config defaults to local Ollama (no API key). To use OpenAI instead, uncomment the open_ai block and set OPENAI_API_KEY.

git clone https://github.com/chaos-substrate/chaos-substrate
cd chaos-substrate
cp chaos-substrate.example.toml chaos-substrate.toml
2
Start Postgres + pgvector 
docker compose up -d

Starts pgvector/pgvector:pg16 on port 54329. For the default Ollama path, ensure Ollama is running and pull the model:

ollama pull nomic-embed-text

Using OpenAI instead? Export your key: export OPENAI_API_KEY=sk-...

3
Build the release binary, run migrations, and verify 
cargo build --release
./target/release/chaos --config chaos-substrate.toml migrate
./target/release/chaos --config chaos-substrate.toml doctor

doctor reports Postgres reachability, pgvector, provider, model name, and embedding dimensions. All five lines must be green before indexing.

4
Index a repository 
./target/release/chaos --config chaos-substrate.toml analyze /path/to/your/repo

Extracts functions, classes, interfaces, types, structs, traits, imports, call edges, Cargo/npm dependencies, Markdown/MDX docs, and PDF context. Produces real embedding vectors for every chunk. Re-run any time to refresh.

5
Query the index 
./target/release/chaos --config chaos-substrate.toml query /path/to/your/repo \
  "where is request validation handled?"

Returns source-grounded chunks with file paths, line ranges, scores, and graph context paths — from durable Postgres memory, not a re-scan of the filesystem.

Add to your editor

One command: chaos setup

Auto-detects Claude Code, Codex, Cursor, Windsurf, and OpenCode and registers chaos-substrate as an MCP server in each. Merges into existing config — never clobbers.

# Preview all planned changes without writing
./target/release/chaos --config chaos-substrate.toml setup --dry-run

# Apply to all detected editors
./target/release/chaos --config chaos-substrate.toml setup

# project scope applies to Claude Code's MCP registration (other editors use their fixed config paths)
./target/release/chaos --config chaos-substrate.toml setup --scope project

Scopes: user (user-level config), local (machine-local), project (project scope via Claude Code's claude mcp add; other editors always use their fixed config paths). Idempotent — rerunning re-writes the chaos-substrate entry with the current values; other MCP servers in the file are preserved.

For Cursor, Windsurf, and OpenCode per-editor JSON paths, and for the full hooks detail, see docs/EDITOR_SETUP.md.

ⓘ Hook support. The plugin ships chaos hook — a safe, no-op hook that injects code-memory context on Grep/Glob/Bash tool calls in Claude Code and Cursor. It always exits 0 and requires no running embedder.

Claude Code — manual

Use the wrapper script (builds binary if needed and calls claude mcp add):

# Machine-local (private)
scripts/chaos-agent claude-code-add local

# Shareable project .mcp.json
scripts/chaos-agent claude-code-add project /abs/path/to/target-repo

# User-level
scripts/chaos-agent claude-code-add user

Or register directly with the Claude Code CLI:

claude mcp add chaos-substrate -- \
  /abs/target/release/chaos \
  --config /abs/chaos-substrate.toml mcp

See docs/CLAUDE_MCP_INSTALL.md for Claude Desktop JSON config and the plugin/skills/hooks flow.

Codex — manual

Register with the Codex CLI:

codex mcp add chaos-substrate -- \
  /abs/target/release/chaos \
  --config /abs/chaos-substrate.toml mcp

Or add directly to ~/.codex/config.toml:

[mcp_servers.chaos-substrate]
command = "/abs/target/release/chaos"
args    = ["--config", "/abs/chaos-substrate.toml", "mcp"]

Replace /abs with the absolute path to your Chaos Substrate checkout. See docs/EDITOR_SETUP.md for Cursor, Windsurf, and OpenCode JSON config blocks.

MCP tools

9 tools — the canonical tool surface

The stdio MCP server speaks newline-delimited JSON-RPC (no Content-Length framing) and exposes exactly nine tools. Agents should prefer these when available; do not synthesize feature pages from chaos_query alone when chaos_feature_context and chaos_write_feature_website are available.

Tool What it does Key parameters When to use
chaos_analyze Indexes or refreshes a repository into the persistent graph + embeddings. Extracts symbols, call edges, imports, dependencies, and document context. Re-running replaces the stored index for that repo. repo_path First, before any query. Re-run after the repo changes.
chaos_add Incrementally indexes only git-diff changed files (working tree by default, a committed range via since, or explicit paths incl. Markdown/exports and PDFs), re-embedding just the changed chunks. Then refreshes the Obsidian vault and writes a feature/bug HTML page — all in one call. Cross-file call edges into unchanged files are not rebuilt; run chaos_analyze for a full graph rebuild. repo_path, paths, since, kind (feature|bug), message, obsidian_output, no_obsidian, no_page After editing a few files, to refresh memory without re-indexing the whole repo.
chaos_stats Reports index statistics for an already-indexed repository, read from Postgres: totals (files, nodes, edges, chunks, embedded vs missing, split chunks, nodes with a chunk) plus breakdowns of nodes by kind, edges by kind, chunks by type, and files by language. Read-only and embedder-free. repo To explain or sanity-check what an chaos_analyze or chaos_add produced.
chaos_query Answers a focused, source-grounded question via hybrid (semantic + keyword) retrieval against the persisted chunks and graph paths. repo, question, limit (default 10) Targeted questions about specific code without re-reading files.
chaos_feature_context Gathers evidence for understanding a feature: semantic/keyword hits, graph context paths, and feature-page manifests from docs/features_memory. Pass output_html to also write the feature page in the same call. repo, task, limit (10), feature_limit (3), nodes_per_feature (8), features_dir, output_html Before implementing or explaining a feature, to assemble an implementation brief.
chaos_impact Builds a feature-vs-existing-code impact report for an indexed repo and always writes an interactive HTML (impact summary + evidence dashboard) to docs/features_memory/<slug>-impact.html. Returns a compact JSON summary — counts plus the existing files/symbols the feature touches, warnings, and the HTML path — keeping the full evidence in the HTML so it won't flood agent context. Frames how a proposed feature maps onto the codebase as it exists today (the before). repo, feature, output_html, features_dir, limit (10), feature_limit (3), nodes_per_feature (8) Before implementing a feature, to see what it touches in the current codebase as a shareable impact page.
chaos_write_feature_website Writes an LLM-composed feature page plus its machine-readable manifest block. The page must include interactive graph/story/code navigation, architecture/flow sections, evidence, and a populated manifest — not a README-style summary. repo, slug, title, html, manifest To persist a reviewed feature explanation as a shareable static page.
chaos_obsidian Exports an already-indexed repository as an Obsidian vault — one Markdown note per graph node, grouped into topic notes, plus an edge manifest — read from the persisted graph. Run after chaos_analyze, which never writes files. repo, output To produce a browsable Markdown vault of the indexed graph; defaults output to <repo>/chaos-obsidian-vault.
chaos_refresh Regenerates project-local artifacts from the persisted index without re-indexing: rewrites the Obsidian vault and, with all_features=true, re-renders the deterministic feature pages in docs/features_memory from their embedded manifests. repo, obsidian_output, features_dir, all_features After chaos_analyze or chaos_add, to rebuild local docs/pages without re-embedding.
Example

Illustrative flow

The following shows a typical agent session after setup. Commands use the CLI mirror of the MCP tools; the MCP equivalents accept the same concepts as JSON parameters.

Step 1 — index the repo

# Via CLI
./target/release/chaos --config chaos-substrate.toml analyze /path/to/my-api

# Via MCP tool (agent call)
# chaos_analyze({ "repo_path": "/path/to/my-api" })

Step 2 — query (illustrative output)

# chaos_query({ "repo": "/path/to/my-api", "question": "where is authentication middleware configured?", "limit": 5 })

# Illustrative grounded response:
# [1] src/middleware/auth.rs:42-78  score=0.91
#     fn require_auth(req: &Request) -> Result<Claims> { ... }
#     → imports: jwt::decode, config::JWT_SECRET
#     → called by: src/routes/api.rs:UserRouter::new
#
# [2] src/config.rs:12-19  score=0.84
#     JWT_SECRET, TOKEN_EXPIRY, REFRESH_WINDOW
#     → depends_on: std::env, dotenvy
ⓘ This is an illustrative example — actual output depends on the indexed repository. All answers include file paths and line ranges from the persisted graph, not generated text.

Step 3 — gather feature context

# chaos_feature_context({
#   "repo": "/path/to/my-api",
#   "task": "implement OAuth2 refresh-token rotation",
#   "limit": 10,
#   "output_html": "docs/features_memory/oauth2-refresh.html"
# })

Returns semantic and keyword hits, graph context paths, and any existing feature manifests from docs/features_memory. With output_html set, also writes the interactive feature page in the same call.

Verify & troubleshoot

Verify the setup

After setup — or any time something seems wrong — run doctor first:

./target/release/chaos --config chaos-substrate.toml doctor

doctor probes Postgres reachability, confirms pgvector is installed and the vector extension is enabled, runs a real embedding probe against the configured provider, and reports the provider name, model, and dimensions. All five checks must pass before indexing.

After indexing, confirm with a sample query:

./target/release/chaos --config chaos-substrate.toml query \
  /path/to/repo "what files are indexed?"

In the editor, confirm all nine MCP tools are listed: chaos_analyze, chaos_add, chaos_stats, chaos_query, chaos_feature_context, chaos_impact, chaos_write_feature_website, chaos_obsidian, chaos_refresh.

Common failures

  • Embedder not configured → analysis fails by design. This is correct behavior. Chaos Substrate will not store fake vectors. Set OPENAI_API_KEY or ensure Ollama is running at http://localhost:11434 with nomic-embed-text pulled. See docs/OLLAMA_SETUP.md.
  • Postgres not reachable. Run docker compose ps and confirm the container is up. The default port is 54329 (not the standard 5432). Check DATABASE_URL in your config or environment.
  • chaos not found in PATH. After cargo build --release, the binary is at target/release/chaos. Use the full absolute path in MCP/editor config, or install it to a directory on your PATH.
  • MCP tools not visible in editor. Restart the editor after modifying MCP config. Run ./target/release/chaos --config chaos-substrate.toml mcp manually to confirm the binary starts without errors.
  • Dimension mismatch error. Switching between OpenAI (1536 dims) and Ollama (768 dims) on the same database without re-indexing will be rejected. Re-run analyze after changing providers to rebuild vectors.
  • graph.html empty or missing nodes. Confirm analyze completed without errors, check nodes/edges table counts in Postgres, and make sure the repo argument matches the indexed absolute path or repository name.
